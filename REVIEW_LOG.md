The `POST /talk` endpoint at `src/claude-agent-server.ts:670-706` is the primary interface for sending messages to a worker agent. It validates the `message` field, generates a unique `queryId` with `Date.now()` + random suffix, persists a `query.received` news item (best-effort — DB failure doesn't block the request), then delegates to `startQuery()` and immediately returns `202 Accepted` with the query ID. Error handling is thorough — empty results, `is_error` JSON payloads, and harness errors are all caught and surfaced. One minor note: the `queryId` uses `Date.now()` which could theoretically collide under extreme load, but the random suffix makes this negligible in practice. No security concerns — authentication is handled by the middleware layer above this route.

## Item 121 — PATCH /identity (worker agent) | Status: REVIEW | 2026-03-24 | Functional but lacks input validation on identity fields

The `PATCH /identity` endpoint at `src/claude-agent-server.ts:1083-1118` allows updating an agent's onchain identity fields: `tokenId`, `registry`, `registry7930`, `metadata`, and `domain`. It merges new fields into the existing `agentIdentity` object via spread operators, with `metadata` being deep-merged one level. The primary caller is the manager's `registerOnchainAndUpdateAgent` flow (agent-manager-db.ts:725), which PATCHes the agent's identity after successful onchain registration. The endpoint correctly rejects empty payloads (400 if no recognized fields), and `setIdentity()` (line 1121) also syncs the `agentName` and catalog from metadata. However, there is **no input validation or type checking** on the accepted fields — `tokenId` could be set to an object, `domain` to a number, or `registry` to any arbitrary JSON. In the current deployment model this is low-risk since the only caller is the trusted manager, but if the agent port were exposed without auth (`ID_REQUIRE_CLIENT_AUTH` defaults to false), any network-reachable client could overwrite the agent's identity to impersonate another agent or set a malicious domain. The `metadata` deep merge is also unbounded — an attacker could inject large payloads into the agent's memory. Recommendation: add type assertions on identity fields (string checks for tokenId/domain/registry7930) and consider requiring auth for this endpoint specifically, similar to how `/health` and `/.well-known/restap.json` are exempted from auth but other endpoints are not.

Overall, a clean and correct implementation.

## Item 136 — Table `news_items` | Status: BUG | 2026-03-24 | Schema is sound but agent-copy query references non-existent columns

The `news_items` table defined at `src/db.ts:253-263` stores the async message feed with columns: `id` (bigserial PK), `team_id` (uuid FK to teams with CASCADE), `agent_id` (text), `timestamp` (bigint), `type` (text), `message` (text), `data` (jsonb), and `query_id` (text). It has a composite FK `(team_id, agent_id)` referencing `agents(team_id, id)` with `ON DELETE CASCADE`, which correctly cleans up news when agents are deleted. Two indexes cover the primary access patterns: `news_items_agent_time_idx` on `(team_id, agent_id, timestamp)` for the `/news` polling endpoint, and `news_items_query_idx` on `(team_id, agent_id, query_id)` for query-specific lookups. The migration path (`src/db.ts:166-186`) safely renames the column through three historical names (`network_id` → `container_id` → `project_id` → `team_id`) using conditional `IF EXISTS` checks, which is robust. All INSERT and SELECT operations across the codebase (`claude-agent-server.ts:291`, `local-agent-server.ts:317`, `interactive-agent-server.ts:102`, `agent-manager-db.ts:1121,1193,4286`) correctly use the actual schema columns. **However**, the agent copy/transfer function at `src/agent-manager-db.ts:2182-2186` contains a **bug**: `INSERT INTO news_items (team_id, agent_id, query_id, result, created_at) SELECT $1, agent_id, query_id, result, created_at FROM news_items WHERE ...` — the columns `result` and `created_at` do not exist in the `news_items` table. This query will throw a PostgreSQL error at runtime whenever an agent is transferred between teams and has existing news items. The correct columns should be `timestamp, type, message, data, query_id`. This bug is latent — it only fires during team-to-team agent transfers.
