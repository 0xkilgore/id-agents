## Item 103 — `DELETE /teams/:name` | Status: REVIEW | 2026-03-25 | Safe agent guard but no protection against deleting the default team, soft-deleted agents silently cascade-deleted

The `DELETE /teams/:name` endpoint at `src/agent-manager-db.ts:1612-1656` deletes a team from the database after verifying it has no active agents. The flow: look up team by name via parameterized query (safe from SQL injection), count agents with `deleted_at IS NULL`, reject with 400 if `agentCount > 0`, then `DELETE FROM teams WHERE id = $1`. The team directory is intentionally preserved on disk (commented-out code at lines 1648-1649 with "keep files as backup" rationale). The CLI side at `interactive-agent-cli.ts:1443-1476` adds a confirmation prompt (`confirmAction`) and switches to the "default" team if the active team was deleted, which is good UX. Two issues worth noting: **No default team protection** — there is no guard against deleting the "default" team, which is the fallback for `getTeamName()` (line 395: `'default'`). If the default team is deleted, any request without an explicit `X-Id-Team` header would trigger `getOrCreateTeamId` to recreate it, but any agents or data in the original default team would be gone. **Soft-deleted agents cascade silently** — the check only counts agents where `deleted_at IS NULL`, so soft-deleted agents (with `deleted_at` set) don't block deletion. When the team row is deleted, `ON DELETE CASCADE` (db.ts:219) hard-deletes all agent rows including soft-deleted ones, plus their associated wallets, news_items, and queries. This is arguably correct behavior (soft-deleted = no longer active), but it means the soft-delete record is permanently lost. The endpoint uses safe parameterized queries throughout, has proper error handling, and the deliberate decision to keep the filesystem directory is sensible for data safety. Overall, functional with minor guardrail gaps.

## Item 93 — `POST /news/archive` | Status: REVIEW | 2026-03-25 | Functional archival but SELECT+DELETE not in a transaction, no bounds on `days` parameter

The `POST /news/archive` endpoint at `src/agent-manager-db.ts:1339-1400` archives news items older than a configurable day threshold to JSON files and deletes them from the database. The flow is: parse `days` from request body (default 30), compute `cutoffTimestamp`, SELECT all matching items, write them to `{baseWorkDir}/teams/{teamName}/archives/news-archive-{date}-{ts}.json`, then DELETE from the database. The implementation has two notable issues. **No transaction**: the SELECT (line 1346) and DELETE (line 1384) use separate `pool.query()` calls without a transaction. If a new news item is inserted between SELECT and DELETE with a timestamp below the cutoff (unlikely but possible with clock skew or backdated inserts), that item would be deleted without being archived. More practically, if `writeFileSync` succeeds but the DELETE query fails (e.g., DB connection drops), the items remain in the database — on retry, the same items get archived again to a new file, creating duplicates. This is a minor data integrity concern, not data loss, since the failure mode is duplication rather than deletion. **No bounds on `days`**: `parseInt(req.body?.days) || 30` accepts any integer — `days: 0` or `days: -1` would compute a cutoff in the future, archiving all or nearly all news items. The CLI side (interactive-agent-cli.ts:2569) also does `parseInt(daysArg) || 30` with no validation. In practice only the CLI operator calls this, so the risk is self-inflicted, but a minimum bound (e.g., `Math.max(1, days)`) would be prudent. The archive directory path uses `teamName` from the `X-Id-Team` header without path sanitization, but this is a systemic issue with `getTeamName()` (line 379) shared across all endpoints, not specific to this one. The file content itself is well-structured with clear metadata (archivedAt, cutoffDays, itemCount). Overall, the endpoint works correctly for its intended use case but would benefit from wrapping the SELECT+DELETE in a transaction.

## Item 78 — `POST /agents/spawn` | Status: REVIEW | 2026-03-25 | Functional agent spawning with minor concerns: triple metadata UPDATE, no name validation, port allocation race

The `POST /agents/spawn` endpoint at `src/agent-manager-db.ts:1734-1896` is the core agent creation path, handling agent ID generation, workspace setup, plugin merging, DB registration, port allocation, and heartbeat initialization. It accepts a rich set of parameters (name, type, model, runtime, plugins, skills, apiKey, claudeMd, heartbeat, etc.) and returns enough metadata for the CLI to spawn the local process. The implementation works but has several areas worth noting. **Triple metadata write**: the handler INSERTs the agent row (line 1825), then immediately UPDATEs metadata to add `agent_account` (line 1834), then UPDATEs again with port/endpoint/metadata (line 1846) — three DB round-trips where one INSERT with the final values would suffice. This is a performance nit, not a bug. **No name validation**: the `name` parameter is checked for presence (`if (!name)`) but not format — names with spaces, special characters, or excessive length are accepted and will propagate to directory names and display IDs. The `workingDirectory` path is constructed as `{baseWorkDir}/agents/{id}` where `id` is system-generated, so name injection doesn't affect filesystem paths, but it could cause display issues or confuse agent resolution. **Port allocation race**: `dbNextPort()` (line 541) finds the max port in the team's range and returns `max + 1`, but this is not atomic — concurrent spawn requests could get the same port. In practice this is unlikely (spawns are typically sequential from the CLI), but under load the INSERT would succeed for both and two agents would attempt to bind the same port. **Error cleanup is partial**: on error (line 1881-1895), the agent's status is set to `'error'` but the workspace directory created at line 1776 is not cleaned up. This is acceptable since orphaned directories are harmless and can be cleaned manually. The `claudeMd` write at line 1785 uses `writeFileSync` directly from user input without size limits, but the Express JSON body parser provides a default 100KB limit. Overall, the endpoint is functional and well-structured for its complexity, but the triple-write pattern and lack of name format validation are worth addressing.

## Item 121 — PATCH /identity (worker agent) | Status: REVIEW | 2026-03-24 | Functional but lacks input validation on identity fields

The `PATCH /identity` endpoint at `src/claude-agent-server.ts:1083-1118` allows updating an agent's onchain identity fields: `tokenId`, `registry`, `registry7930`, `metadata`, and `domain`. It merges new fields into the existing `agentIdentity` object via spread operators, with `metadata` being deep-merged one level.
